164  * INTERPLUGIN MESSAGING

166 /*
167  * Plugin messages are defined as 32-bit integers. Messages below 0x00FFFFFF
168  * are reserved for X-Plane and the plugin SDK.
169  *
170  * Messages come with a pointer parameter; the meaning of this pointer depends
171  * on the message itself. In some messages, the pointer parameter contains an
172  * actual typed pointer to data that can be inspected in the plugin; in these
173  * cases the documentation will state that the parameter "points to"
174  * information.
175  *
176  * in other cases, the value of the pointer is actually an integral number
177  * stuffed into the pointer's storage. In these second cases, the pointer
178  * parameter needs to be cast, not dereferenced. In these caess, the
179  * documentation will state that the parameter "contains" a value, which will
180  * always be an integral type.
181  *
182  * Some messages don't use the pointer parameter - in this case your plugin
183  * should ignore it.
184  *
185  * Messages have two conceptual uses: notifications and commands. Commands
186  * are sent from one plugin to another to induce behavior; notifications are
187  * sent from one plugin to all others for informational purposes. It is
188  * important that commands and notifications not have the same values because
189  * this could cause a notification sent by one plugin to accidentally induce a
190  * command in another.
191  *
192  * By convention, plugin-defined notifications should have the high bit set
193  * (e.g. be greater or equal to unsigned 0x8000000) while commands should have
194  * this bit be cleared.
195  *
196  * The following messages are sent to your plugin by X-Plane.
197  *
198  */
199

295 /***************************************************************************
296  * Plugin Features API
297  ***************************************************************************/
298 /*
299  * The plugin features API allows your plugin to "sign up" for additional
300  * capabilities and plugin system features that are normally disabled for
301  * backward compatibility. This allows advanced plugins to "opt-in" to new
302  * behavior.
303  *
304  * Each feature is defined by a permanent string name. The feature string
305  * names will vary with the particular installation of X-Plane, so plugins
306  * should not expect a feature to be guaranteed present.
307  *
308  * XPLM_WANTS_REFLECTIONS
309  * ----------------------
310  *
311  * Available in the SDK 2.0 and later for X-Plane 9, enabling this capability
312  * causes your plugin to receive drawing hook callbacks when X-Plane builds
313  * its off-screen reflection and shadow rendering passes. Plugins should
314  * enable this and examine the dataref sim/graphics/view/plane_render_type to
315  * determine whether the drawing callback is for a reflection, shadow
316  * calculation, or the main screen. Rendering can be simlified or omitted for
317  * reflections, and non-solid drawing should be skipped for shadow
318  * calculations.
319  *
320  * **Note**: direct drawing via draw callbacks is not recommended; use the
321  * XPLMInstance API to create object models instead.
322  *
323  * XPLM_USE_NATIVE_PATHS
324  * ---------------------
325  *
326  * available in the SDK 2.1 and later for X-Plane 10, this modifies the plugin
327  * system to use Unix-style paths on all operating systems. With this enabled:
328  *
329  * * OS X paths will match the native OS X Unix.
330  * * Windows will use forward slashes but preserve C:\ or another drive letter
331  * when using complete file paths.
332  * * Linux uses its native file system path scheme.
333  *
334  * Without this enabled:
335  *
336  * * OS X will use CFM file paths separated by a colon.
337  * * Windows will use back-slashes and conventional DOS paths.
338  * * Linux uses its native file system path scheme.
339  *
340  * All plugins should enable this feature on OS X to access the native file
341  * system.
342  *
343  * XPLM_USE_NATIVE_WIDGET_WINDOWS
344  * ------------------------------
345  *
346  * Available in the SDK 3.0.2 SDK, this capability tells the widgets library
347  * to use new, modern X-Plane backed XPLMDisplay windows to anchor all widget
348  * trees. Without it, widgets will always use legacy windows.
349  *
350  * Plugins should enable this to allow their widget hierarchies to respond to
351  * the user's UI size settings and to map widget-based windwos to a VR HMD.
352  *
353  * Before enabling this, make sure any custom widget code in your plugin is
354  * prepared to cope with the UI coordinate system not being th same as the
355  * OpenGL window coordinate system.
356  *
357  */
358
