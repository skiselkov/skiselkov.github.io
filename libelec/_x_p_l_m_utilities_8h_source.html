23 /*
24  * The XPLMUtilities file APIs provide some basic file and path functions for
25  * use with X-Plane.
26  *
27  * Directory Separators
28  * --------------------
29  *
30  * The XPLM has two modes it can work in:
31  *
32  * * X-Plane native paths: all paths are UTF8 strings, using the unix forward
33  * slash (/) as the directory separating character. In native path mode,
34  * you use the same path format for all three operating systems.
35  *
36  * * Legacy OS paths: the directroy separator is \ for Windows, : for OS X,
37  * and / for Linux; OS paths are encoded in MacRoman for OS X using legacy
38  * HFS conventions, use the application code page for multi-byte encoding
39  * on Unix using DOS path conventions, and use UTF-8 for Linux.
40  *
41  * While legacy OS paths are the default, we strongly encourage you to opt in
42  * to native paths using the XPLMEnableFeature API.
43  *
44  * * All OS X plugins should enable native paths all of the time; if you do
45  * not do this, you will have to convert all paths back from HFS to Unix
46  * (and deal with MacRoman) - code written using native paths and the C
47  * file APIs "just works" on OS X.
48  *
49  * * For Linux plugins, there is no difference between the two encodings.
50  *
51  * * Windows plugins will need to convert the UTF8 file paths to UTF16 for
52  * use with the "wide" APIs. While it might seem tempting to stick with
53  * legacy OS paths (and just use the "ANSI" Windows APIs), X-Plane is fully
54  * unicode-capable, and will often be installed in paths where the user's
55  * directories have no ACP encoding.
56  *
57  * Full and Relative Paths
58  * -----------------------
59  *
60  * Some of these APIs use full paths, but others use paths relative to the
61  * user's X-Plane installation. This is documented on a per-API basis.
62  *
63  */
64

487 /***************************************************************************
488  * X-PLANE COMMAND MANAGEMENT
489  ***************************************************************************/
490 /*
491  * The command management APIs let plugins interact with the command-system in
492  * X-Plane, the abstraction behind keyboard presses and joystick buttons. This
493  * API lets you create new commands and modify the behavior (or get
494  * notification) of existing ones.
495  *
496  * X-Plane Command Phases
497  * ----------------------
498  *
499  * X-Plane commands are not instantaneous; they operate over a duration.
500  * (Think of a joystick button press - you can press, hold down, and then
501  * release the joystick button; X-Plane commands model this entire process.)
502  *
503  * An X-Plane command consists of three phases: a beginning, continuous
504  * repetition, and an ending. The command may be repeated zero times in its
505  * duration, followed by one command ending. Command begin and end messges are
506  * balanced, but a command may be bound to more than one event source (e.g. a
507  * keyboard key and a joystick button), in which case you may receive a second
508  * begin during before any end).
509  *
510  * When you issue commands in the plugin system, you *must* balance every call
511  * to XPLMCommandBegin with a call to XPLMCommandEnd with the same command
512  * reference.
513  *
514  * Command Behavior Modification
515  * -----------------------------
516  *
517  * You can register a callback to handle a command either before or after
518  * X-Plane does; if you receive the command before X-Plane you have the option
519  * to either let X-Plane handle the command or hide the command from X-Plane.
520  * This lets plugins both augment commands and replace them.
521  *
522  * If you register for an existing command, be sure that you are *consistent*
523  * in letting X-Plane handle or not handle the command; you are responsible
524  * for passing a *balanced* number of begin and end messages to X-Plane. (E.g.
525  * it is not legal to pass all the begin messages to X-Plane but hide all the
526  * end messages).
527  *
528  */
529
