13 /*
14  * ## THEORY OF OPERATION AND NOTES
15  *
16  * Widgets are persistent view 'objects' for X-Plane. A widget is an object
17  * referenced by its opaque handle (widget ID) and the APIs in this file. You
18  * cannot access the widget's guts directly. Every Widget has the following
19  * intrinsic data:
20  *
21  * - A bounding box defined in global screen coordinates with 0,0 in the
22  * bottom left and +y = up, +x = right.
23  * - A visible box, which is the intersection of the bounding box with the
24  * widget's parents visible box.
25  * - Zero or one parent widgets. (Always zero if the widget is a root widget.
26  * - Zero or more child widgets.
27  * - Whether the widget is a root. Root widgets are the top level plugin
28  * windows.
29  * - Whether the widget is visible.
30  * - A text string descriptor, whose meaning varies from widget to widget.
31  * - An arbitrary set of 32 bit integral properties defined by 32-bit integral
32  * keys. This is how specific widgets store specific data.
33  * - A list of widget callbacks proc that implements the widgets behaviors.
34  *
35  * The Widgets library sends messages to widgets to request specific behaviors
36  * or notify the widget of things.
37  *
38  * Widgets may have more than one callback function, in which case messages
39  * are sent to the most recently added callback function until the message is
40  * handled. Messages may also be sent to parents or children; see the
41  * XPWidgetDefs.h header file for the different widget message dispatching
42  * functions. By adding a callback function to a window you can 'subclass' its
43  * behavior.
44  *
45  * A set of standard widgets are provided that serve common UI purposes. You
46  * can also customize or implement entirely custom widgets.
47  *
48  * Widgets are different than other view hierarchies (most notably Win32,
49  * which they bear a striking resemblance to) in the following ways:
50  *
51  * - Not all behavior can be patched. State that is managed by the XPWidgets
52  * DLL and not by individual widgets cannot be customized.
53  * - All coordinates are in global screen coordinates. Coordinates are not
54  * relative to an enclosing widget, nor are they relative to a display
55  * window.
56  * - Widget messages are always dispatched synchronously, and there is no
57  * concept of scheduling an update or a dirty region. Messages originate
58  * from X-Plane as the sim cycle goes by. Since X-Plane is constantly
59  * redrawing, so are widgets; there is no need to mark a part of a widget as
60  * 'needing redrawing' because redrawing happens frequently whether the
61  * widget needs it or not.
62  * - Any widget may be a 'root' widget, causing it to be drawn; there is no
63  * relationship between widget class and rootness. Root widgets are
64  * imlemented as XPLMDisply windows.
65  *
66  */

79 /*
80  * XPCreateWidget
81  *
82  * This function creates a new widget and returns the new widget's ID to you.
83  * If the widget creation fails for some reason, it returns NULL. Widget
84  * creation will fail either if you pass a bad class ID or if there is not
85  * adequate memory.
86  *
87  * Input Parameters:
88  *
89  * - Top, left, bottom, and right in global screen coordinates defining the
90  * widget's location on the screen.
91  * - inVisible is 1 if the widget should be drawn, 0 to start the widget as
92  * hidden.
93  * - inDescriptor is a null terminated string that will become the widget's
94  * descriptor.
95  * - inIsRoot is 1 if this is going to be a root widget, 0 if it will not be.
96  * - inContainer is the ID of this widget's container. It must be 0 for a root
97  * widget. for a non-root widget, pass the widget ID of the widget to place
98  * this widget within. If this widget is not going to start inside another
99  * widget, pass 0; this new widget will then just be floating off in space
100  * (and will not be drawn until it is placed in a widget.
101  * - inClass is the class of the widget to draw. Use one of the predefined
102  * class-IDs to create a standard widget.
103  *
104  * A note on widget embedding: a widget is only called (and will be drawn,
105  * etc.) if it is placed within a widget that will be called. Root widgets are
106  * always called. So it is possible to have whole chains of widgets that are
107  * simply not called. You can preconstruct widget trees and then place them
108  * into root widgets later to activate them if you wish.
109  *
110  */
111 WIDGET_API XPWidgetID XPCreateWidget(
112  int inLeft,
113  int inTop,
114  int inRight,
115  int inBottom,
116  int inVisible,
117  const char * inDescriptor,
118  int inIsRoot,
119  XPWidgetID inContainer,
120  XPWidgetClass inClass);
