25 /***************************************************************************
26  * Terrain Y-Testing
27  ***************************************************************************/
28 /*
29  * The Y-testing API allows you to locate the physical scenery mesh. This
30  * would be used to place dynamic graphics on top of the ground in a plausible
31  * way or do physics interactions.
32  *
33  * The Y-test API works via probe objects, which are allocated by your plugin
34  * and used to query terrain. Probe objects exist both to capture which
35  * algorithm you have requested (see probe types) and also to cache query
36  * information.
37  *
38  * Performance Guidelines
39  * ----------------------
40  *
41  * It is generally faster to use the same probe for nearby points and
42  * different probes for different points. Try not to allocate more than
43  * "hundreds" of probes at most. Share probes if you need more. Generally,
44  * probing operations are expensive, and should be avoided via caching when
45  * possible.
46  *
47  * Y testing returns a location on the terrain, a normal vectory, and a
48  * velocity vector. The normal vector tells you the slope of the terrain at
49  * that point. The velocity vector tells you if that terrain is moving (and is
50  * in meters/second). For example, if your Y test hits the aircraft carrier
51  * deck, this tells you the velocity of that point on the deck.
52  *
53  * Note: the Y-testing API is limited to probing the loaded scenery area,
54  * which is approximately 300x300 km in X-Plane 9. Probes outside this area
55  * will return the height of a 0 MSL sphere.
56  *
57  */
58

299 /*
300  * XPLMLoadObject
301  *
302  * This routine loads an OBJ file and returns a handle to it. If X-Plane has
303  * already loaded the object, the handle to the existing object is returned.
304  * Do not assume you will get the same handle back twice, but do make sure to
305  * call unload once for every load to avoid "leaking" objects. The object will

308  * The path for the object must be relative to the X-System base folder. If
309  * the path is in the root of the X-System folder you may need to prepend ./

311  * discouraged - your plugin should not dump art resources in the root folder!
312  *
313  * XPLMLoadObject will return NULL if the object cannot be loaded (either
314  * because it is not found or the file is misformatted). This routine will
315  * load any object that can be used in the X-Plane scenery system.
316  *
317  * It is important that the datarefs an object uses for animation already be
318  * loaded before you load the object. For this reason it may be necessary to
319  * defer object loading until the sim has fully started.
320  *
321  */

352  * XPLMDrawObjects
353  *
354  * __Deprecation Warning__: use XPLMInstancing to draw 3-d objects by creating
355  * instances, rather than these APIs from draw callbacks.
356  *
357  * XPLMDrawObjects draws an object from an OBJ file one or more times. You
358  * pass in the object and an array of XPLMDrawInfo_t structs, one for each
359  * place you would like the object to be drawn.
360  *
361  * X-Plane will attempt to cull the objects based on LOD and visibility, and
362  * will pick the appropriate LOD.
363  *
364  * Lighting is a boolean; pass 1 to show the night version of object with
365  * night-only lights lit up. Pass 0 to show the daytime version of the object.
366  *
367  * earth_relative controls the coordinate system. If this is 1, the rotations
368  * you specify are applied to the object after its coordinate system is
369  * transformed from local to earth-relative coordinates -- that is, an object
370  * with no rotations will point toward true north and the Y axis will be up
371  * against gravity. If this is 0, the object is drawn with your rotations from
372  * local coordanates -- that is, an object with no rotations is drawn pointing
373  * down the -Z axis and the Y axis of the object matches the local coordinate
374  * Y axis.
375  *
376  */
