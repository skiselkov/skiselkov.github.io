13 /*
14  * Plug-ins can create menus in the menu bar of X-Plane. This is done by
15  * creating a menu and then creating items. Menus are referred to by an
16  * opaque ID. Items are referred to by (zero-based) index number.
17  *
18  * Menus are "sandboxed" between plugins---no plugin can access the menus of
19  * any other plugin. Furthermore, all menu indices are relative to your
20  * plugin's menus only; if your plugin creates two sub-menus in the Plugins
21  * menu at different times, it doesn't matter how many other plugins also
22  * create sub-menus of Plugins in the intervening time: your sub-menus will be
23  * given menu indices 0 and 1. (The SDK does some work in the back-end to
24  * filter out menus that are irrelevant to your plugin in order to deliver
25  * this consistency for each plugin.)
26  *
27  * When you create a menu item, you specify how we should handle clicks on
28  * that menu item. You can either have the XPLM trigger a callback (the
29  * XPLMMenuHandler_f associated with the menu that contains the item), or you
30  * can simply have a command be triggered (with no associated call to your
31  * menu handler). The advantage of the latter method is that X-Plane will
32  * display any keyboard shortcuts associated with the command. (In contrast,
33  * there are no keyboard shortcuts associated with menu handler callbacks with
34  * specific parameters.)
35  *
36  * Menu text in X-Plane is UTF8; X-Plane's character set covers latin, greek
37  * and cyrillic characters, Katakana, as well as some Japanese symbols. Some
38  * APIs have a inDeprecatedAndIgnored parameter that used to select a
39  * character set; since X-Plane 9 all localization is done via UTF-8 only.
40  *
41  */
